## 三个核心原则

- **以读者为中心**：先想"谁会读、带着什么问题读"，再决定写什么。
- **场景化**：用一个具体场景代替三段抽象描述。
- **可执行**：每段话读完，读者应该知道下一步做什么。

## 好文档 vs 坏文档

**坏文档的样子**

- 列了 API 字段，却不说怎么调用
- 说"请参考下文"，却不给跳转链接
- 大段抽象描述，没有一个例子

**好文档的样子**

- 给一个最小可运行的代码示例
- 引用精准到行号，能点击跳转
- 用具体场景说明"什么时候用、什么时候别用"

## 一段最小示例胜过千言

文档里最该出现的，是这种「贴上就能跑」的代码：

```bash
curl -X POST https://api.example.com/v1/users \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"name":"Alice"}'
```

读者把 `$TOKEN` 替换成自己的就能验证——比十段"如何认证"的描述都管用。

## 三个常见误区

> [!WARNING]
> **追求完整性**：试图覆盖所有边界情况，结果谁都没耐心看完。

> [!WARNING]
> **只更新代码不更新文档**：6 个月后文档已经全是骗人的。

> [!WARNING]
> **闭门造车**：写完不让任何人试读就上线，结果新人看了一头雾水。

## 工具对比

挑工具的优先级：**能用就好，别让选型本身成为不写文档的借口。**

| 工具         | 适合场景          | 学习成本 |
|------------|---------------|------|
| Markdown   | 任何项目，跟代码一起版本化 | 低    |
| Notion     | 团队协作、内部知识库    | 中    |
| GitBook    | 对外的产品文档       | 中    |
| Docusaurus | 大型开源项目站点      | 高    |

## 小结

> [!IMPORTANT]
> 文档不是"写完就完事"的一次性产物，而是和代码一样需要**持续维护、迭代、贴近用户**——开始写、坚持改、让人真的用上，比一开始就选对工具重要得多。
