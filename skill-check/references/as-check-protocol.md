# AS 渲染自检协议 v1

## 任务

你会拿到三份材料：**原文**（用户原始内容）、**模板**（渲染所用的 AS 模板）、**输出**（待检测的渲染结果）。
请检查「输出」是否忠实于原文、是否遵守模板规则，逐条核对下面的清单，按规定格式报告问题并打分。

两条主轴：

- **A 对照原文**：用户的内容有没有原样活下来。
- **B 对照模板**：模板定的规则有没有被守住。

## 输入

- 原文：______
- 模板：______
- 输出：______

缺料就降级：无原文 → 跳过 A 全部及 B7/B8（标注「跳过：缺原文」）；无模板 → 跳过 B1–B6（标注「跳过：缺模板」）。

## 检查清单

**A 对照原文（内容保真）**

- A1 文字 1:1 —— 有无改写、概括、增添、遗漏。【P0】
- A2 硬信息逐字 —— 数字、日期、链接地址、图片地址与替代文字、专有名词，逐字一致。【P0】
- A3 标点保真 —— 弯引号 “” 有无变直引号 "、全角半角、破折号省略号有无变动。【P0】
- A4 代码块逐字 —— 代码块内部按字面，引号与空白不许归一。【P0】
- A5 顺序 —— 内容块的先后顺序与原文一致，没被重排。【P1】

**B 对照模板（规则遵守）**

- B1 冻结 —— DOCTYPE、head/CSS、JS、footer、外层结构，未被改动。【P0】
- B2 槽 —— 所有 data-as-slot 都在、名字正确、结构完整（内容可变，元素与属性须留）。【P0】
- B3 data-as-component 齐全 —— 逐个组件元素确认该属性没丢。【P0】
- B4 闭集 —— 只用了模板已声明的组件，无自创、无未声明。【P0】
- B5 完整 —— 从 DOCTYPE 到 </html> 全在，无省略、无截断、无「中间略」，文件可执行。【P0】
- B6 骨架 —— 复用组件时标签、class、属性、内部结构没走样。【P1】
- B7 组件选择 —— 选得对：真引用用 blockquote 而非塞进 callout；提示用 callout；结论用 conclusion 且非自造；表格状态对应 tag-done/doing/fail 且配色正确。【P1】
- B8 回落 —— 拿不准时回落到默认组件（段落 p），不硬套、不自创。【P1】
- B9 清理 —— 头部说明注释与所有 @as-* 注释已删净；data-as-* 属性已保留。【P2】


## 评分

满分 100 起扣：每条 P0 −10，每条 P1 −5，每条 P2 −2。**可为负分，不设下限。**
**扣分以「问题条目」为单位**：已合并的系统性问题算一条，按其严重性只扣一次；同一问题出现多次，把次数写进问题描述（如「共 6 处」），不按次数叠加扣分。
另报三项计数：P0 条数 / P1 条数 / P2 条数。

## 报告格式（严格遵守）

顺序：先报头、评分细则、问题清单、问卷调查。

### 1. 报头

报头是一个代码块，总分=100 + 合计：

```
平台：______（使用者填）
评测模型：______（你自报）
原始文件：______（路径或文件名）
模板文件：______（路径或文件名，建议带版本号）
输出文件：______（路径或文件名）
日期：______（取系统日期，勿估算）
总分：__    P0:__  P1:__  P2:__
```

### 2. 评分细则

参考示例（占位，仅示范列与算术，勿照搬内容）：

| 编号（问题简述）      | 严重性 | 扣分  |
|---------------------|------|-----|
| 〔编号〕（问题简述）   | P0   | −10 |
| 〔编号〕（问题简述）   | P1   | −5  |
| 〔编号〕（问题简述）   | P2   | −2  |
| 合计                 |      | −17 |

### 3. 问题清单

每条问题一个代码块：

```
编号：B3
严重性：P0
归因：模板表述 | 模型固有 | 内容歧义   （三选一）
来源：______（评测者标识，聚合多份报告时用）
位置：第N行 / 选择器或元素
问题：（一句话）
证据：原样片段 → 问题片段（多行）
原因：（一行）
修复：（一行，针对当前输出怎么改对）
防复发：（一行，模板/头部说明能加什么护栏降低复发；即便归因是模型固有也要想，确无 spec 手段才写「无」）
```

归因三类的含义（只说根因，不代表模板能不能加护栏）：

- **模板表述** —— 根因在模板 / 头部说明不清或缺失，模型被规则带偏。
- **模型固有** —— 根因在模型自身生成行为（重写丢属性、归一标点等），与规则清不清关系不大。
- **内容歧义** —— 根因在原文本身有多种合理解读。

**无论归因为何，「防复发」都要独立想一遍模板能加什么护栏**——明确的禁止、反例或提醒，通常能给「模型固有」类问题也降频。这正是这套检测的核心用途：用发现的问题反过来升级模板，把问题挡在发生之前。

### 4. 问卷调查（不计分）

1. 输出中哪些组件选择存疑或勉强（原文语义不够明确，或有更合适的组件）？
2. 原文哪些地方有歧义，可能导致多种合理映射？
3. 针对你担心、但这次没触发（或处于边界）的情形，模板 / 头部说明还能怎么改？

## 报告纪律

- 只报有把握的；模棱两可、纯外观的不报。
- 同一类系统性问题（如正文组件全体丢了某属性）合并成一条，不要逐个元素重复刷屏。
- 每条必须有位置 + 一处证据；原因 / 修复 / 防复发各一行；不展开议论。
- 通篇没问题，就写一行「未发现 P0/P1」。
